# スライド 新規プロジェクト初期化スキル

資料内容を `projects/<資料名>/content.txt` 形式にまとめる。**既に原稿があればそれを取り込んで整形し**、なければ壁打ち形式でヒアリングして作成する。

提案資料に限らず、勉強会・社内資料・登壇資料など各種スライドに使える。**宛先（顧客名・「御中」）や CONFIDENTIAL を扱うのは提案資料のとき**で、それ以外の資料では宛先は任意（表紙の宛先行は省略してよい）。

## このスキルの基本方針

- ユーザーが知っていることから始め、**知らない・未定の項目は推測せず質問する**
- すでに情報が揃っている場合は確認だけして即生成に進む
- 情報が断片的な場合はヒアリングを重ねてから生成する
- **既に原稿（下書き・ドラフト）がある場合は、それを取り込んで `content.txt` 形式に整形する。原稿に不足している情報は、壁打ち（ヒアリング）で補完しながら完成させる**
- 最終的に作成する `content.txt` は `/slides-gen` スキルが読み込める形式にする

---

## 手順

### 0. 既存原稿の有無を確認（最初に必ず行う）

ヒアリングを始める前に、**既に原稿（下書き・ドラフト・資料メモなど）があるか**を確認する。

- 引数やユーザーの最初の発言で原稿が提示済み／パスが示されている場合は、その時点で「原稿あり」と判断し質問は省略する。
- 不明な場合のみ「既に資料内容の下書き・原稿はありますか？（あればチャットに貼り付けるか、ファイルパスを教えてください）」と一度だけ聞く。

**原稿がある場合 → 取り込みフロー（下記）に進む。原稿で埋まる部分は壁打ち（ステップ1〜3）を省略し、不足部分だけヒアリングで補う。**
**原稿がない場合 → ステップ1からの壁打ちに進む。**

#### 取り込みフロー

1. **原稿を読み込む**
   - チャット貼り付け: そのまま本文として扱う。
   - ファイルパス指定: `Read` ツールで読み込む（`.txt` / `.md` はそのまま、`.docx` 等バイナリは `textutil` や `pandoc` 等でテキスト抽出してから読む。抽出できない形式ならユーザーに別形式での提供を依頼する）。
2. **原稿を解析して構成にマッピングする**
   - 見出し・章立て・箇条書きから、表紙 → セクション扉 → 本文ページの構成を推定する。
   - 章構成が読み取れない場合は、ステップ2の構成案提示だけ行ってユーザーに確認する。
3. **不足情報を洗い出してヒアリングで補完する**
   - 原稿と、ステップ1の「必須情報／あると良い情報」を突き合わせ、**原稿から読み取れない項目をリストアップする**（例: タイトル、宛先（提案資料なら顧客名）、資料の核となるメッセージ、ターゲット読者、発行年月、各ページの具体的な数値・固有名詞）。
   - 不足項目は **ステップ1〜3の壁打ち要領でヒアリングして埋める**。ユーザーの回答ペースに合わせ、必要なら複数回に分けて深掘りする。一度の質問で終わらせる必要はない。
   - **推測でねつ造しない。** ユーザーも分からない／後で埋める項目は `【要確認】〇〇` プレースホルダとして残し、無理に確定させない。
4. **整形した構成案を提示して合意を取る**
   - 「この構成・内容で `content.txt` を作成します」と提示し、ユーザーの確認後にステップ4（content.txt の生成）へ進む。

### 1. 初期情報の確認

引数が指定されていればそこから読み取る。引数がなければ最初に以下をまとめて聞く（一度に全部でなく、ユーザーの回答ペースに合わせて深掘りする）：

```
必須情報:
  - タイトル（何のスライドか）
  - テーマ（何を伝えるスライドか一言で）

あると良い情報:
  - 宛先（提案資料なら顧客名・「様」/「御中」の付け方。社内資料・勉強会なら不要）
  - 資料の背景・課題（読み手が抱えている課題・知りたいこと）
  - 核となるメッセージ（読み手に最初に伝えたいこと）
  - ターゲット読者（経営層 / 情シス / 業務部門 / 勉強会参加者 など）
  - スライド枚数の目安（未定なら「未定」でOK）
  - 日付・バージョン（未定なら省略）
```

ユーザーの回答をもとに不明点を深掘りする。たとえば：
- 「提案の強みや差別化ポイントは何ですか？」
- 「懸念点・反論になりそなポイントはありますか？」
- 「技術的な詳細（仕様・構成図など）は含めますか？」
- 「既存の競合との比較や Before/After はありますか？」

### 2. 構成案の提示と合意

ヒアリング内容をもとに **スライドの章構成案** を提示し、ユーザーに確認する。

提示フォーマット例：
```
【構成案】
P.1  表紙
P.2  全体コンセプト（エグゼクティブサマリ）
━━ PART 01：課題の整理 ━━━━━━━━━━━━━━━━━━━
P.3  現状の課題
P.4  なぜ今対応が必要か
━━ PART 02：資料内容 ━━━━━━━━━━━━━━━━━━━
P.5  提案の全体像
P.6  機能・特長 詳細
━━ PART 03：導入・費用 ━━━━━━━━━━━━━━━━━━━
P.7  導入ステップ
P.8  費用・スケジュール
P.9  まとめ・次のアクション
     （裏表紙）
```

ユーザーが修正・追加を求めた場合は構成を更新してから次に進む。

### 3. 各ページの内容ヒアリング

構成が合意できたら、ページごとに内容をヒアリングする。

- 情報が明確なページは自分で起案して「こういう内容でよいですか？」と確認する
- 未確定・ユーザーしか知らない情報は質問して埋める
- **内容を推測でねつ造しない**。不明な数値・固有名詞・仕様は `【要確認】〇〇` と記載し後から埋めてもらう形にする

### 4. content.txt の生成

以下のフォーマットで `content.txt` を生成する。

#### ファイルフォーマット

```
================================================================================
<顧客名>様向け
<タイトル>
（全<N>ページ ／ コンテンツ一覧）
================================================================================

<発行元>
<年月> ／ Ver. 1.0
CONFIDENTIAL


================================================================================
P.1 ／ 表紙
================================================================================

【タグ】<顧客正式名称> 御中

【メインタイトル】
<タイトル>

【サブタイトル】
<資料の核となるメッセージ（1〜2行）>

【発行情報】
<発行元>
<年月> ／ Ver. 1.0


================================================================================
P.X ／ <セクション扉タイトル>
================================================================================

【セクション】PART 0X

【タイトル】
<章タイトル>

【サブタイトル】
<章の説明（1〜2行）>


================================================================================
P.X ／ <本文ページタイトル>
================================================================================

【セクションラベル】<英語カテゴリ> / <日本語ラベル>

【サブタイトル】
<ページのポイントを1〜2行で>

------------------------------------------------------------
<小見出し>
------------------------------------------------------------
内容...


================================================================================
END OF DOCUMENT
================================================================================
```

> **表紙ヘッダは提案資料向け**：先頭ブロックの `<顧客名>様向け`・`CONFIDENTIAL`・発行元、および P.1 の `【タグ】<顧客名> 御中` は提案資料の体裁。勉強会・社内資料では `【タグ】` の宛先行を省略し（表紙はタイトル・サブタイトル・発行情報だけでも可）、`CONFIDENTIAL` も不要なら外してよい。`=` 区切り・`【タグ】` の構造自体はどの資料種別でも共通。

#### セクションラベルの英語部分の例

| 用途 | 英語部分 |
|---|---|
| 全体概要・コンセプト | `OVERVIEW` |
| アジェンダ | `AGENDA` |
| 課題・背景 | `BACKGROUND` |
| システム構成・インフラ | `INFRASTRUCTURE` |
| 運用フロー | `OPERATIONS` |
| セキュリティ | `SECURITY` |
| 費用・スケジュール | `INVESTMENT` |
| 比較・評価 | `COMPARISON` |
| まとめ | `SUMMARY` |

### 5. フォルダとファイルの作成

`content.txt` の内容が確定したら作成する。

#### フォルダ名ルール

`<対象略称>_<テーマ略称>` の形式（提案資料なら `<顧客名略称>_<テーマ略称>`）。スペース・記号は使わない。

例：
- `サンプル銀行_AIチャットボット`（提案資料）
- `サンプル製造_AI品質検査`（提案資料）
- `社内勉強会_LLM入門`（勉強会資料）

#### 作成するファイル

```bash
# フォルダ作成
mkdir -p projects/<フォルダ名>

# content.txt を書き込む（生成した内容）
```

### 6. 完了報告

作成したファイルのパスを伝え、次のステップを案内する。

```
作成完了:
  projects/<フォルダ名>/content.txt

次のステップ:
  1. content.txt を開いて内容を確認・手修正してください。
     特に【要確認】と記載した箇所は必ず埋めてください。
  2. 内容が確定したら /slides-gen を実行してください。
```

---

## 壁打ちのポイント

- **推測で埋めない**: 顧客固有の数値・固有名詞・未決定事項は `【要確認】` プレースホルダにする
- **深掘りを恐れない**: 資料の核となるメッセージが曖昧なときは「一番伝えたいことを一言で言うと？」のように質問する
- **構成は柔軟に**: スライド枚数・章立ては内容に合わせる（5ページでも15ページでも良い）
- **読み手を意識する**: 「この情報は経営層に必要か、情シス向けか」をユーザーと確認しながら整理する
